# Deploying Warden to Cloudflare Pages

The landing site (`site/`) is a zero-dependency static site. It deploys to
**Cloudflare Pages** with no build step and gets a free `*.pages.dev` HTTPS URL
immediately — perfect for going live before you buy a custom domain. The
trust-graded registry is served from the same project at `/registry/`.

> Two one-line edits first (see [`site/README.md`](site/README.md)): set
> `WAITLIST_ENDPOINT` in `site/app.js`, and replace the `chadcorp/warden`
> placeholder URLs. Then regenerate the registry page: `py -m warden build-index`.

## What gets deployed

```
site/                     ← publish directory (everything below is static)
├── index.html            the landing page
├── styles.css  app.js    assets
├── favicon.svg  og-image.svg
├── _headers  _redirects  Cloudflare Pages config
└── registry/             the trust-graded index (generated by `warden build-index`)
    ├── index.html        human-browsable
    └── index.json        signed, machine-verifiable
```

## Option A — Dashboard (Git-connected, recommended)

1. Push this repo to GitHub/GitLab.
2. Cloudflare dashboard → **Workers & Pages → Create → Pages → Connect to Git**.
3. Pick the repo. Build settings:
   - **Framework preset:** None
   - **Build command:** *(leave empty)*
   - **Build output directory:** `site`
4. **Save and Deploy.** You get `https://<project>.pages.dev` on HTTPS in ~30s.
5. Every push redeploys. (Re-run `py -m warden build-index` and commit when the
   registry changes, so `site/registry/` stays current.)

## Option B — Wrangler CLI (direct upload, no Git)

```bash
npm i -g wrangler          # one-time
wrangler login             # opens your Cloudflare login in the browser
wrangler pages deploy site --project-name warden
```

This uploads `site/` and returns the `*.pages.dev` URL. No build, no framework.

> You run these commands and approve the login — they touch your Cloudflare
> account, so they're yours to execute, not something automated for you.

## The free `.dev`-style URL now, custom domain later

- **Now:** the `*.pages.dev` subdomain is live on HTTPS immediately — share it,
  collect waitlist signups, gather GitHub stars. No purchase required.
- **Later (custom `.dev` domain):** buy e.g. `warden.dev`, then in the Pages
  project → **Custom domains → Set up a domain** → add it; Cloudflare issues the
  certificate automatically. Note the `.dev` TLD is on the HSTS preload list, so
  it is **HTTPS-only by design** — which Cloudflare Pages already serves, so
  there's nothing extra to configure.

## Security headers

`site/_headers` ships a sane Content-Security-Policy and the usual hardening
(`nosniff`, `frame-ancestors 'none'`, a strict referrer policy). The CSP allows
the waitlist `fetch` over HTTPS; tighten `connect-src https:` to your exact
form endpoint when you wire it.

## Hosting the Scan API (later, separate)

The landing site is static and belongs on Pages. The **Scan API**
(`py -m warden serve-api`) is a Python service and does **not** run on Pages
(Pages is static + JS/WASM Workers). When you're ready to host it, run it on any
small VM/container (Fly.io, Railway, a VPS) behind HTTPS, and point a subdomain
(e.g. `api.warden.dev`) at it. That's a Phase-3 step; the local API works today.
